# 0002 — Task 7: frontend PWAs (`apps/creator`, `apps/learner`)

**Covers:** turning `apps/creator`/`apps/learner` into real, installable Next.js PWAs sharing
`packages/ui` and `packages/types`, per ADR-012.

## What was built

- **`apps/creator`** (port 3001) and **`apps/learner`** (port 3002) — real Next.js 16 App Router
  apps, each with a placeholder route (`Creator Dashboard`, `Lesson View`), a PWA manifest
  (`app/manifest.ts`, native Next.js file convention, served at `/manifest.webmanifest`), and a
  minimal passthrough service worker (`public/sw.js`) registered client-side. Learner stays
  deliberately thin — no builder/dashboard code, per ADR-012's "learner bundle stays genuinely
  thin" — verified via a creator-only marker component absent from learner's production build.
- **`packages/ui`** gained its first real component, `Badge` (`{ label: string }`, plain server
  component). Ships as raw `.tsx` source (`main`/`types` point at `src/index.ts`) — no build
  step; Next's bundler transpiles workspace-linked source directly. `react`/`react-dom` are
  `peerDependencies`, not `dependencies`.
- **`packages/types`** gained five real domain enums from ADR §3 (`ProgramShape`,
  `ProgramVisibility`, `LessonType`, `EnrollmentState`, `SignalType`), as `as const`
  string-literal-union objects rather than the `enum` keyword — these values cross a
  serialization boundary (HTTP between `apps/api` and the two frontends, eventually Prisma/JSON),
  and `as const` objects ARE the JSON value with zero extra runtime cost.
- PWA icons: real PNGs (192×192, 512×512) generated by a zero-dependency script
  (`apps/*/scripts/gen-placeholder-icons.mjs`, uses `node:zlib` to hand-assemble PNG chunks) —
  see "Bugs found" below for why SVG-only wasn't sufficient.

## Why

ADR-012: two separately deployed, separately bundled PWAs (not a native app, not a unified
frontend) — creators need a dense builder/dashboard surface, learners need a light,
infrequent-use surface with most engagement happening over WhatsApp.

## Bugs found and fixed

Three genuine surprises this time, on top of the SVG/PNG installability finding below.

### 1. Missing DOM lib broke `next build`'s type check

**Symptom:** `next build` failed with `'navigator.serviceWorker' is of type 'unknown'` inside
`ServiceWorkerRegister.tsx`.

**Root cause:** the shared `packages/config/tsconfig.base.json` only sets `"lib": ["ES2022"]`
(correct for a Node/NestJS-oriented base) — no DOM types, so `navigator` wasn't typed at all in
any app extending it without an override.

**Fix:** added `"lib": ["ES2022", "DOM", "DOM.Iterable"]` to both `apps/creator` and
`apps/learner`'s `tsconfig.json` overrides.

**Lesson:** when deliberately choosing not to copy a framework's generated config wholesale (the
plan explicitly avoided a blind `create-next-app` tsconfig copy in favor of a minimal override
set), double-check every field that's actually load-bearing — `lib` looked like boilerplate but
wasn't.

### 2. `eslint-plugin-react`'s auto-detection crashes on ESLint 10 flat config

**Symptom:** `pnpm --filter creator lint` (and learner) crashed outright — not a normal lint
failure — with `TypeError: contextOrFilename.getFilename is not a function`, on any `.tsx` file.

**Root cause:** `eslint-config-next` pulls in `eslint-plugin-react@7.37.5`, whose React-version
auto-detection calls an ESLint `Linter` API (`context.getFilename()`) that no longer exists in
ESLint 10's flat-config runtime.

**Fix:** added an explicit `settings: { react: { version: '19.2.7' } }` block to
`packages/config/eslint/next.js`, which skips the crashing auto-detection code path entirely.

**Lesson:** this crash was only found because the full-pipeline sanity check
(`pnpm lint`/`pnpm build` from root) ran `apps/creator`/`apps/learner`'s `lint` script for the
first time — manual boot-testing (dev server + curl) never exercises `lint` at all. Always run
every script that CI will run, not just the ones your manual verification happens to touch.

### 3. SVG-only manifest icons don't satisfy PWA installability

**Symptom/finding (not a crash, a design decision made concrete):** the original plan considered
whether a hand-authored SVG icon alone (`sizes: "any"`, `type: "image/svg+xml"`) would satisfy
Chrome/Edge's "Install app" prompt requirements.

**Conclusion:** no — Chromium's installability audit expects at least one raster (PNG) icon at
192×192 and/or 512×512; SVG-only icon sets have historically been flagged by Lighthouse's
"Installable manifest" check. Real PNGs are required.

**Approach taken:** rather than add an image-processing dependency (`sharp`, `jimp`, etc.) just
to generate two solid-color placeholder squares, wrote a ~90-line zero-dependency PNG encoder
using Node core's `zlib.deflateSync` (`apps/*/scripts/gen-placeholder-icons.mjs`) — hand-assembles
the PNG chunk structure (`IHDR`/`IDAT`/`IEND`, including a from-scratch CRC32 table). Validated
independently with the `file` command (`PNG image data, 192 x 192, 8-bit/color RGBA,
non-interlaced` — correct on all four generated files). Kept the SVG too (harmless, satisfies
some icon contexts) but the PNGs are what actually matters for installability.

**Note:** `sharp` did end up getting installed anyway — it's a transitive dependency Next.js
itself uses optionally for image optimization, unrelated to our icon generation. Its
`[ERR_PNPM_IGNORED_BUILDS]` prompt was approved the same way `esbuild`/`unrs-resolver` were in
earlier tasks.

## How to extend / verify

- **Add a new placeholder route:** create `apps/creator/app/<route>/page.tsx` (or learner's
  equivalent) following the App Router convention; import shared UI from `@fundi/ui`, shared
  enums/types from `@fundi/types`.
- **Add a new shared component to `packages/ui`:** export it from `src/index.ts`; if it uses
  React hooks, note that `packages/ui` currently stays on `@fundi/config/eslint/base` (no
  `react-hooks/*` rules yet — see `packages/ui/README.md`) and revisit that gap at that point.
- **Verify PWA installability:** `pnpm --filter creator dev`, then `curl localhost:3001/manifest.webmanifest`
  and `curl localhost:3001/sw.js` should both return valid content; real browser installability
  needs an actual Chrome/Edge session (not verifiable from this environment — curl+grep against
  rendered HTML/served assets was used as the closest available proxy).
- **Verify bundle isolation still holds:** `pnpm --filter creator build && pnpm --filter learner build`,
  then grep both `.next` output trees for any marker string unique to creator-only code — must be
  absent from learner's.
- **Re-run the whole pipeline:** `pnpm lint`, `pnpm build` from root (`@fundi/docs` is correctly
  excluded — no `build`/`lint` script, confirmed by real `pnpm build`/`pnpm lint` runs showing
  "6 successful, 6 total" out of 7 packages in scope, not 7).
